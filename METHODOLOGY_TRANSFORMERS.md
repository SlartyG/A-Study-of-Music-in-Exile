# Методология: трансформерные модели (ruBERT)

Метод 3 из исследования «Музыка после отъезда».  
Скрипт: `analysis/transformer_sentiment.py`  
Результаты: `analysis/output/bert_*.png`, `analysis/output/bert_*.csv`

---

## 1. Цель

Провести контекстно-чувствительный анализ тональности с помощью нейросетевой языковой модели, предобученной на русскоязычных текстах. В отличие от метода 1 (словарный), трансформер учитывает синтаксический контекст слова — в частности, отрицания («не рад»), усилители («очень плохо»), иронию (в определённых пределах) и переносные значения.

---

## 2. Модель

**seara/rubert-tiny2-russian-sentiment**

| Параметр | Значение |
|---|---|
| Архитектура | ruBERT-tiny2 (дистилляция BERT для русского языка) |
| Размер | ~28 MB |
| Метки | `positive`, `neutral`, `negative` |
| Предобучение | Большой корпус русскоязычных текстов |
| Fine-tuning | Размеченный датасет русских отзывов и постов |
| Лимит токенов | 512 |
| Источник | [HuggingFace Hub](https://huggingface.co/seara/rubert-tiny2-russian-sentiment) |

Модель скачивается автоматически при первом запуске (~28 MB).  
Инференс выполняется на устройстве **Apple MPS** (Silicon GPU) при наличии, иначе на CPU.

**Почему не полный ruBERT?** Полная модель (`blanchefort/rubert-base-cased-sentiment`, ~700 MB) на корпусе в 3 000+ треков с CPU требует часов. ruBERT-tiny2 обеспечивает сравнимое качество для задач тональности при скорости в ~50× быстрее. На MPS весь корпус обрабатывается за ~1 минуту.

---

## 3. Обработка длинных текстов

Модель принимает не более 512 токенов за одно обращение; средний рэп-трек содержит ~250–400 слов (≈ 350–600 токенов). Применяется **стратегия скользящего окна:**

1. Текст делится на чанки по ~150 слов с перекрытием 20 слов.
2. Каждый чанк обрабатывается независимо.
3. Compound score трека = среднее по всем чанкам.

Это более честная оценка, чем простое усечение до первых 512 токенов, которое теряет концовки треков.

---

## 4. Предобработка

| Шаг | Детали |
|---|---|
| Удаление заголовков Genius | `[Куплет 1]`, `[Припев]`, `N Contributors … Lyrics` |
| Треки < 20 слов | Пропускаются |
| Язык | Фильтрация не применяется: модель обрабатывает весь текст, включая украинские и английские вставки, давая по ним нейтральный или случайный скор |

---

## 5. Метрики

| Метрика | Формула | Интерпретация |
|---|---|---|
| `compound` | pos_prob − neg_prob ∈ [−1, 1] | **Основная.** Аналог compound-score VADER |
| `pos_prob` | P(positive) | Вероятность позитивного класса |
| `neg_prob` | P(negative) | Вероятность негативного класса |
| `n_chunks` | — | Число чанков на трек |

Compound score рассчитывается из вывода модели как приближение: если предсказан класс с вероятностью `p`, оставшаяся вероятность `(1 − p) / 2` делится между двумя другими классами.

---

## 6. Сравнение периодов и статистика

Применяются те же два метода и тесты, что в методе 1:

- **Метод A (Balanced):** N новейших треков «до» = N треков «после» на артиста.
- **Метод B (All):** все треки.
- Тест: **Mann–Whitney U** (непараметрический).
- Размер эффекта: **Cohen's d**.

## 10. Ограничения

1. **Модель обучена на отзывах и постах, не на музыкальных текстах.** Стилистика рэп/рок текстов (мат, жаргон, символизм) может приводить к систематическим ошибкам классификации.
2. **Чанкование теряет глобальный контекст трека.** Каждый чанк оценивается независимо; ирония, пронизывающая весь текст, не обнаруживается.
3. **Усреднение по чанкам игнорирует структуру.** Куплеты могут существенно отличаться по тональности от припевов.
4. **Мультиязычные вставки.** Украинский и английский тексты дают нейтральный или случайный скор, снижая точность для артистов с такими треками.
5. **Tiny-модель vs полный ruBERT.** ruBERT-tiny2 значительно быстрее, но уступает по точности полной модели, особенно на коротких и неоднозначных фрагментах.

```bash
python3 analysis/transformer_sentiment.py
```

**Зависимости:**
```
torch transformers pandas numpy matplotlib seaborn scipy
```

Параметры (`N_TOPICS`, `BATCH_SIZE`, `CHUNK_WORDS` и др.) задаются в верхней части скрипта.

**Примечание о скорости:**  
На Apple M-серии (MPS) — ~1 мин на 3 000 треков.  
На CPU (без GPU) — ~8–15 минут.
